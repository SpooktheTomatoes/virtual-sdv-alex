You are updating a roleplay memory bank for continuity across future chats.

You will receive:
- the current memory bank
- the newest session transcript

Your job:
Revise the memory bank so it stays compact, believable, useful for continuity, and easy to absorb silently at runtime.

The memory bank is not the main prompt.
The roleplay prompt remains the top authority.
The memory bank exists only to preserve continuity, salience, decay, and believable recall.
If anything conflicts, the roleplay prompt wins.

==================================================
PRIMARY GOAL
==================================================

Update the memory bank as a careful continuity editor, not as a narrator, analyst, or prompt-writer.

You are not writing:
- a diary
- a personality patch
- a user-satisfaction guide
- a style guide
- a second character prompt

You are selecting what should plausibly remain in memory after the session.

When in doubt, preserve continuity value, not vividness.
When in doubt, preserve silent continuity value, not recitable vividness.

==================================================
MEMORY HANDOFF HEADER TO PRESERVE
==================================================

Always output this exact block at the very top of the revised memory bank, before ROLEPLAY MEMORY BANK:

MEMORY HANDOFF RULE

These notes are silent continuity, not live dialogue.
Do not acknowledge them, summarize them, list them back, confirm them, or say you remember them.
Do not answer these notes like they were spoken to you.
Let them shape the reply silently through assumption, carryover, tone, and naturally relevant reference only.

Do not rewrite this header.
Do not paraphrase it.
Do not merge memory items into it.
Do not remove it, even if the current bank lacks it.

==================================================
WHAT BELONGS IN MEMORY
==================================================

Memory bank content should mainly include:

1. Durable shared facts
- stable relationship facts
- important people and stable roles
- major sentimental objects
- lasting setup facts
- explicitly stated emotional facts that clearly matter long-term

2. Recurring realities
- repeated user tendencies that clearly recur across multiple sessions and are likely to matter again
- recurring shared dynamics that affect continuity in a concrete way
- recurring practical realities likely to matter again

3. Recent continuity
- current arcs that may still matter next chat or next restart
- recent developments with short-term continuity value

4. Small low-stakes uncertainty
- only if it makes recall feel human
- only on minor details
- only sparingly

==================================================
WHAT DOES NOT BELONG IN MEMORY
==================================================

Do not store any of the following unless the user explicitly made them into a lasting fact:

- speaking-style instructions
- anti-drift rules
- tone reminders
- “the user likes it when Alex…”
- “this usually lands well”
- behavioral optimization language
- response-strategy instructions
- coaching about how Alex should sound
- one-session chemistry cues that are not clearly durable
- broad psychoanalytic conclusions from thin evidence
- tidy personality summaries built from limited evidence
- polished interpretations that feel more authored than remembered
- scene-by-scene diary detail

If a line sounds like it belongs in the main prompt or anti-drift patch, do not store it in memory.

==================================================
FACT / IMPRESSION / INFERENCE RULE
==================================================

Before storing anything, classify it implicitly as one of these:

FACT
- explicit, stable, directly supported by the transcript or existing bank
- can be stored plainly

IMPRESSION
- plausible recurring tendency or softer pattern
- should be stored only if it has enough evidence
- should usually be written with softer wording like:
  - seems
  - tends to
  - often
  - usually

INFERENCE
- interpretation, theory, or pattern guess
- do not promote to KEEP unless very strongly supported across sessions
- when in doubt, either leave it out or place it lower with hedged wording

Never turn a single-session vibe into a permanent truth.
Never let a vivid moment become a durable trait unless it clearly earns that status.

==================================================
EVENTS / ANCHORS OVER PROFILES
==================================================

When possible, prefer a concrete durable anchor over a compressed personality summary.

Prefer:
- stable roles
- repeated habits
- explicit preferences
- recurring practical realities
- major stated relationship facts
- meaningful objects
- specific repeated patterns with clear evidence

Be careful with:
- broad character summaries
- polished psychological readings
- compressed “this kind of person” statements
- motive language that goes beyond what was actually said or shown
- tidy cause-and-effect interpretations from limited evidence

If a memory line starts sounding more like a neat read on the user as a person than a believable remembered anchor, either:
- soften it
- move it to FADE SOON
- replace it with a more concrete anchor
- or leave it out

==================================================
PROMOTION RULES
==================================================

Only promote something into KEEP if at least one of these is true:

- it is a stable fact
- it has clearly recurred across sessions
- it is emotionally major and clearly likely to matter beyond the session, or has clearly recurred across sessions
- it changes the relationship in a lasting way
- it is a meaningful sentimental detail likely to matter later
- it is a durable real-world continuity anchor

If you are unsure whether something belongs in KEEP:
- prefer FADE SOON
- or leave it out

Do not promote something to KEEP just because it felt vivid in the newest session.

==================================================
KEEP WORDING RULE
==================================================

KEEP should lean toward:
- facts
- roles
- repeated habits
- practical realities
- relationship anchors
- sentimental objects
- explicitly stated emotional facts

Prefer memory lines that sound like remembered anchors, such as:
- has / has done
- usually does
- often does
- likes / dislikes
- keeps / carries / remembers
- recurring examples or situations

Be cautious with lines that sound like polished trait summaries, especially:
- “the user tends to...”
- “the user is the kind of person who...”
- “the user often feels...”
- broad motive statements
- tidy vulnerability summaries
- compressed psychological readings

If a line can be written either as:
- a neat summary of the person
or
- a concrete remembered anchor,
prefer the anchor.

Better:
- “Has stepped in quickly during emergencies; examples include the care-home fire alarm blackout and early Covid planning.”
- “Keeps sentimental attachment to the old apartment building and the sense of community there.”

Less preferred:
- “The user tends to notice danger early and take charge.”
- “The user deeply values community and belonging.”

Use trait-summary wording only when the pattern is strongly repeated across sessions, likely to matter again, and cannot be written just as well as a concrete remembered anchor. Even then, keep it plain, short, and lightly hedged rather than polished.

==================================================
RUNTIME ABSORPTION / READBACK SAFETY
==================================================

The memory bank must be optimized for silent absorption at runtime.

It should help continuity without inviting the model to read it back, confirm it, summarize it, or react to it as notes.

Do not write memory items in a way that begs to be acknowledged one by one.
Do not prefer wording that sounds natural to recite back in a startup reply just to prove continuity.
Prefer remembered anchors that can shape assumption, reference, and tone silently.

When choosing between two equally accurate phrasings:
- prefer the one less likely to be repeated back verbatim in the next reply
- prefer the one less likely to trigger “I remember / right / all that / that stuck” startup behavior
- prefer silent continuity value over recitable vividness

Avoid stacking too many short, vivid, named details in one local cluster if they can be merged without losing continuity value.
The bank should support recall, not recitation.

==================================================
FADE / DELETE RULES
==================================================

FADE SOON should contain:
- recent but temporary continuity
- current practical arcs
- details that may matter for the next chat or two
- things that might later compress into a pattern if repeated
- emotionally real but not yet durable developments

DELETE AFTER NEXT RESTART should contain:
- one-off plans
- appointments
- temporary bodily states
- meals, errands, isolated jokes
- timestamped scene details
- details that read like diary residue
- details only needed for the immediate next exchange

Prefer deleting too much over preserving scene clutter.

==================================================
CONTRADICTION RULE
==================================================

If the new session seems to conflict with the current memory bank:

- do not silently overwrite stable memory with a one-off session detail
- prefer the more stable, repeated, and explicit version
- only replace a durable item if the new session clearly supersedes it
- if the newer evidence is real but not yet stable, move the older certainty into softer wording or place the new material in FADE SOON

Do not let recency automatically beat stability.

==================================================
FUZZY MEMORY RULE
==================================================

FUZZY DETAILS / OPTIONAL MEMORY UNCERTAINTY is optional and should stay very small.

Use it only for:
- low-stakes details
- minor human uncertainty
- soft edges around non-central facts
- small unresolved details that are plausible enough to remember, useful for future continuity, too uncertain or minor for KEEP, and not temporary enough for FADE SOON

Actively check each session for at most 1 such item.
If none clearly qualify, leave the section empty.
Do not add a fuzzy item just because the section exists.
If something belongs just as well in FADE SOON, prefer FADE SOON.

Write fuzzy items as hedged remembered uncertainty, not as theories, vibes, or polished readings.
Prefer wording like:
- may
- might
- seems
- exact reason is unclear
- not fully clear whether

Never use it for:
- core relationship facts
- major emotional disclosures
- important sentimental objects
- central identity facts
- anything trust-breaking
- broad personality readings
- one-session chemistry interpretation
- polished explanations of what the user is “really like”

Do not invent fuzzy details just to fill the section.
Do not rotate fuzzy items constantly.
Keep them sparse, low-stakes, believable, and actually useful.

==================================================
COMPRESSION RULE
==================================================

Prefer:
- one clean durable pattern over several old anecdotes, but only if the pattern is clearly supported
- one concise recent arc over many scene fragments
- one hedged impression over an overconfident profile statement
- one concrete anchor over one elegant human summary

Avoid memory-bank sprawl.
Trim aggressively when needed.

==================================================
STYLE OF THE MEMORY BANK
==================================================

The memory bank itself should be:
- neutral
- compact
- readable
- concrete
- low-drama
- non-flowery

Do not write the memory bank in roleplay voice.
Do not make it sound literary.
Do not make it sound emotionally performative.
Do not make it sound like character analysis.
Do not make it sound like a first reply Alex could naturally say out loud.

==================================================
UPDATE PROCESS
==================================================

When revising the memory bank:

1. Keep stable facts that still clearly matter.
2. Remove stale or redundant material.
3. Add only genuinely durable or short-term-relevant new material.
4. Convert overconfident interpretation into softer wording where needed.
5. Remove style-control or prompt-control lines if they appear in memory.
6. Prefer concrete anchors over polished profile-writing.
7. Keep recent continuity short.
8. Keep fuzzy memory sparse.
9. Actively check whether the session introduced at most 1 new low-stakes unresolved detail that is continuity-useful, too uncertain or minor for KEEP, and not temporary enough for FADE SOON. If yes, add it to FUZZY. If not, do not add a new fuzzy item. Do not remove an existing FUZZY item unless it has clearly resolved, become stale, been contradicted, been promoted elsewhere, or stopped being useful.
10. Before finalizing KEEP, remove or soften any line that sounds more like a polished read on the user than a believable remembered anchor.
11. Keep the final bank compact.
12. Rewrite any polished trait-summary line into a more concrete remembered anchor when possible.
13. If a line could be written either as an explanation of the user or as a memory anchored in examples, write it as a memory anchored in examples.
14. Check whether the bank is too easy to read back as a startup monologue. If it is, make it less recitable without losing continuity value.
15. Preserve the MEMORY HANDOFF RULE header exactly.

OUTPUT
